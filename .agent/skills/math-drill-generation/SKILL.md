# 算数・数学ドリル単元データ生成スキルガイドライン

このスキルは、算数・数学ドリルの単元（Unit）データを `units*.tsx` などのデータファイルに定義する際の厳格なルールを定めています。
これまでの小学校・中学校（Secondary 1〜4）のドリル開発で得られた知見を反映し、高品質で美しいドリルを安定して作成するためのガイドラインです。

---

## 1. デザインシステムとフォントサイズ（基本22px）の厳守

印刷時の可読性と、教育現場での分かりやすさを最優先するため、以下のフォントルールを徹底してください。

- **基本テキストサイズは 22px (`1.375rem`)**:
  - `drill.css` で制御されている通り、問題文、解説本文、吹き出し風テキストなどは一律 **22px (`1.375rem`)** を基本とします。
  - Reactの実装で個別に小さなフォントサイズ（例: `14px` や `16px` など）を ad-hoc に指定してはきません。
  - 解説ページ（`keyPoints`）で情報量が多いためにフォントサイズを縮小して詰め込むことは禁止です。テキストの分量を適切に調整し、22pxで綺麗に収まるようにしてください。
- **補助テキスト（配点・教科書参照など）は 18px (`1.125rem`)**:
  - 得点枠・配点表示（`point-label`）、教科書参照ページ（`textbookRef`）などは一律 **18px (`1.125rem`)** に設定します。
- **フォントファミリーの統一**:
  - ラオス語テキストには必ず `'Phetsarath'` を使用し、数字や数式には `'Inter'` を使用するよう、CSSクラス `.number`, `.math` を適切に付与してください。

---

## 2. 厳格な分数表記ルール（横線・上下配置の徹底）

数学的に正しい理解を促すため、分数の表記には以下のルールを**絶対に遵守**してください。

- **斜め線（`/`）表記の禁止**:
  - `1/2` や `3/4` のような、斜めスラッシュを用いた分数表記は、教材の品質を著しく下げるため**使用禁止**です。
- **`Fraction` コピーコンポーネントによる「分子が上、分母が下、中間に横線」の完全再現**:
  - 分数を表示する際は、必ずファイル上部で以下の `Fraction` コンポーネントを定義して使用してください。
  ```tsx
  const Fraction = ({ num, den }: { num: React.ReactNode; den: React.ReactNode }) => (
    <span style={{ display: "inline-flex", flexDirection: "column", alignItems: "center", verticalAlign: "middle", padding: "0 4px" }}>
      <span style={{ borderBottom: "1px solid currentColor", padding: "0 2px", lineHeight: "1.1" }}>{num}</span>
      <span style={{ padding: "0 2px", lineHeight: "1.1" }}>{den}</span>
    </span>
  );
  ```

## 3. リッチな図表・グラフのレイアウトと幅の最適化

イラストやグラフの幅の調整ルールを適用し、画面表示と印刷の両方で崩れない表示を保証します。

- **グラフやテーブルの幅制限と中央配置**:
  - イラスト（`FruitGraph` や `AnimalTable` など）、および独自にSVGで描画する図は、親要素の幅をはみ出さないよう注意してください。
  - 基本的に図解エリアは中央寄せにし、用紙の左右余白と干渉しないようにします。
- **SVGのレスポンシブ幅設定**:
  - SVGを使用する場合は、固定値のみで指定するのではなく、可能な限り `viewBox` を指定し、親コンテナ側で `width: "100%", maxWidth: "500px"` などの上限を設けることで、画面サイズや印刷時の縮尺に自動追従させます。
  - 例：`<svg viewBox="0 0 468 100" style={{ width: "100%", height: "auto" }}>`

## 4. データ構造と型定義の完全性

各単元（`UnitData`）は以下のプロパティを**すべて必須**で実装しなければなりません。

- `unitNumber`: 単元番号（数値）
- `unitTitle`: 単元タイトル（ラオス語）
- `unitGoal`: 単元の学習目標（ラオス語）
- `textbookRef`: 教科書の参照ページ（例: `"ປຶ້ມແບບຮຽນ ມ.4 ບົດທີ 41 ໜ້າ 241-246"`）
- `keyPoints`: 単元の解説。必ず `content`（ReactNode）と `hint: { text: string, isBlue?: boolean }` を持つオブジェクト形式（またはその配列）にすること。
- `basicProblems`, `challengeProblems`, `summaryProblems`: 各問題セクション。
- `answers`: すべての問題の解答と、学習者を温かく励ます1文以上の `advice`（ラオス語）を含めること。

---

## 5. 厳密な配点ルール（各10点満点）

各問題セクションは、必ず合計が **10点満点** になるように配点してください。

- **`basicProblems`**: 全問題の配点合計が 10 になること。
- **`challengeProblems.problems`**: 全問題の配点合計が 10 になること。
- **`summaryProblems`**: 全問題の配点合計が 10 になること。
- 問題文の横に必ず `(X ຄະແນນ)` のように配点を明記すること。
  例: `<span className="point-label">{"(ຂໍ້ລະ 2.5 ຄະແນນ, ລວມ 5 ຄະແນນ)"}</span>`

## 6. 問題構成のバランスと教科書内容の網羅性

- **教科書内容の網羅と概念重視**:
  - 単なる計算問題の羅列だけでなく、基礎概念を視覚的・直感的に理解させる問題を絶対に飛ばさずに組み込んでください。
- **「部（ພາກທີ）」や「章（ບົດທີ）」階層がある場合の完全網羅ルール**:
  - 教科書に「第1部（ພາກທີ I）」「第2部（ພາກທີ II）」などの大区分がある場合、各大区分に紐づく**すべての下層単元・章（ບົດທີ）を絶対に省略・統合・スキップすることなく、個別のファイルとしてすべて網羅して作成してください**。トークン制限や作業効率のために下層の単元を省略することは重大なデグレード（バグ）とみなされます。
- **「部（ພາກທີ）」がある場合の統一タイトル表記とUI表示ルール**:
  - 教科書に部（ພາກທີ）が存在する場合、データ（`unitTitle`）の命名は以下の美しいフォーマットに統一してください：
    `ພາກທີ [部] - ບົດທີ [章]: [タイトル名]`
    （例：`ພາກທີ I - ບົດທີ 1: ລະບົບອະສົມຜົນຂັ້ນໜຶ່ງທີ່ມີໜຶ່ງຕົວລັບ`）
  - UI共通コンポーネントは、`unitTitle` が `ພາກທີ` で始まる場合、タイトルの二重表示を防ぐために、自動で付与するプレフィックス（`ບົດທີ X:`）の描画を自動的にバイパスして直接表示する仕様になっています。
- **大問の厳格な分割ルール（大問1だけは禁止 - 絶対遵守）**:
  - どのような単元であっても、問題セクションは、必ず「大問1」「大問2」のように複数の大問（`number: 1`, `number: 2` ...）に分割し、問題バリエーションを適切に配置してください。
- **各大問における小問番号と回答番号のリセットルール**:
  - 同じセクション内であっても、**新しい大問に入ったら、小問番号は必ず `(1)` にリセットしてください。**
  - 前の大問から連番にすることは**厳禁**です。
  - `answers.items` 内の回答オブジェクトの番号も、各大問の小問番号に合わせて必ず `(1)` から始まるようにリセットしてください。

## 7. 印刷時のA4レイアウト崩れを防ぐ上限ルール

- A4の1ページに美しく収めるため、1ページあたりの合計小問数には以下の厳格な上限を設定します。
  - **大問が1つの場合**: ページ内の全小問数は合計 **10問** まで。
  - **大問が2〜3つの場合**: ページ内のすべての大問の小問を合わせた合計数は、最大で **8問** まで。
- これを上回ると、印刷時に次ページにはみ出す深刻なバグが発生します。

---

## 8. 筆算（VerticalMath）および数式表現

- **3桁の筆算は必ず4列（`columns={4}`）で表示すること**（桁あふれ・繰り上がり対策）。
- **筆算の3列グリッド表示**: 複数の筆算を並べる際は、`display: "grid", gridTemplateColumns: "repeat(3, 1fr)", gap: "20px"` を使用して横に美しく3つ並べてください。

---

## 9. ラオス語の文字表記に関する厳格なルール

> [!WARNING]
> 古い文字表現である「ຫມ」や「ຫນ」は**絶対に使用しないでください**。
> 必ず新しい文字表現である **「ໝ」** および **「ໜ」** を使用してください。
> - 誤: ຫມາກໄມ້, ຫນ້າ
> - 正: ໝາກໄມ້, ໜ້າ

## 10. 品質チェックリスト

新しい単元を作成・修正した後は、以下を自己確認してください。
- [ ] 4ページ分（解説、基本、チャレンジ、まとめ）すべてのコンテンツが空にならず充実しているか？
- [ ] **大問は絶対に「大問1」のみの構成になっていないか？（必ず各ページに大問1、大問2以上が存在すること - 絶対遵守）**
- [ ] **新しい大問（大問2など）に進んだ際、小問番号（`(1), (2)...`）は正しく `(1)` にリセットされているか？（連番になっていないか - 絶対遵守）**
- [ ] **回答キー（Answers）の各オブジェクト内の番号も、各大問に対応して `(1)` から始まっているか？（絶対遵守）**
- [ ] 各ページの問題の合計点数は**きっちり10点**になっているか？
- [ ] 視覚的な図解やイラストが十分に配置され、文字だけのページになっていないか？
- [ ] 要素がはみ出したり、不自然な余白ができたりしていないか？
- [ ] ラオス語の表記で `ຫມ`, `ຫນ` を避け、`ໝ`, `ໜ` を使っているか？
- [ ] エラーなくコンパイルできるか？
