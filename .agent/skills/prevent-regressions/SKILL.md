# 回帰防止と品質チェックガイドライン (Prevent Regressions)

本ガイドラインは、これまでの開発経験（シンタックス破損、TypeScript型不整合、印刷時のレイアウト崩れ、GAS開発など）に基づき、不具合の回帰（デグレード）を防ぎ、品質を最高水準に維持するための手順とチェックリストをまとめたスキルです。

## クイックスタート (事前検証チェックリスト)

コードの編集を行う前、および行った後に以下のチェックリストを必ず実行してください。

- [ ] **最小限の置換範囲**: `replace_file_content` 等を使用する際、壊れるリスクを抑えるため置換範囲を最小限にする。
- [ ] **静的解析の実行**: 編集後すぐに `npx tsc --noEmit` で TypeScript の型不整合を検知する。
- [ ] **ビルドチェック**: `npm run build` を走らせてプロジェクト全体が正常にコンパイル・ビルドされるか確認する。
- [ ] **印刷・表示幅の検証**: レイアウト変更時に要素が 700px 前後に収まるか、重なりや見切れがないか確認する。

## ワークフロー (詳細手順)

### 1. 自動置換時のシンタックス破損防止
非英語文字（ラオス語、日本語など）や複雑な JSX 階層を含むファイルをツールで置換する際、文字エンコーディングや空白・特殊文字の差分により、置換対象が見つからなかったり、隣接するブロックが破損したりするリスクがあります。

* **対策**:
  1. 置換対象の `StartLine` と `EndLine` を極限まで狭め、ピンポイントで置換する。
  2. 置換前に対象コードを `view_file` で再確認し、完全一致する文字列を指定する。
  3. 置換を実行した直後、必ずローカルのビルドチェックを走りらせて構文エラーがないか確認する。

### 2. TypeScript 型定義の不整合検知
データ構造やコンポーネントの Props 定義（例：`UnitData` の `keyPoints` 定義の変更など）を修正すると、他のファイル（`units2.tsx`, `units3.tsx`, `DrillBook.tsx` など）で芋づる式に型エラーが発生する原因になります。

* **対策**:
  * 一箇所の型を変更した場合は、必ず関連する他のファイルに定義されている同等の型インターフェースも確認する。
  * `npx tsc --noEmit` コマンドでプロジェクト全体の静的型チェックを行い、エラーが 0 件であることを確認してからコミットする。

### 3. A4印刷時の表示崩れ・見切れの防止
算数ドリルはA4サイズでの印刷や高解像度の表示を前提としています。要素の横幅やフォントサイズが不適切だと、印刷時に右端が見切れたり、不必要な改ページ（白紙ページ）が発生します。

* **対策**:
  * **グリッド・フレックス**: 横並びにする際は `flexWrap: "nowrap"` または幅に応じた適切なパーセンテージ幅を適用する。
  * **数値の丸め**: 数直線やグラフで動的な数値を扱う場合、JavaScript の浮動小数点数誤差（例: `0.30000000000000004`）によって文字列長が伸びて重なるのを防ぐため、必ず `Math.round(v * 1000) / 1000` などの丸め処理を入れる。
  * **筆算レイアウト**: 筆算が複数並ぶ箇所は、画面幅を考慮して列数（`columns`）やグリッドの列設定（`repeat(3, 1fr)` など）を美しく調整する。

### 4. GAS (Google Apps Script) 連携開発のベストプラクティス
スプレッドシートや管理用GAS APIと通信を行うシステム開発では、認証やデータスキーマの変化に起因するバグが発生しやすい傾向があります。

* **対策**:
  * API レスポンスやスプレッドシートの入力データを処理する箇所には、必ず `try-catch` ブロックによる例外処理を実装し、エラー時も画面が真っ白にならないようにフォールバック（代替表示）を用意する。
  * テスト環境と本番環境のスプレッドシートIDや環境変数を完全に分離して管理する。
